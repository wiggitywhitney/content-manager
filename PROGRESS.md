# Progress Log

Development progress log for content-manager. Tracks implementation milestones across PRD work.

Entry format: `- (YYYY-MM-DD) Description of feature-level change (PRD #X, milestone)`

## [Unreleased]

### Fixed
- (2026-06-18) Fixed two bugs that caused gist posts (board photo rows) to fail on Bluesky and micro.blog. `postToMicroblog` was unconditionally calling `extractYouTubeVideoId` on the `youtubeUrl` field, which is a board image URL (not YouTube) for gist rows — crashing before the post was attempted. Fixed by accepting an optional `imageBuffer` parameter: when a pre-fetched buffer is provided, the YouTube video ID extraction is skipped entirely and the buffer is used directly as the post image. Bluesky was rejecting board PNG images exceeding its 2,000,000-byte upload limit. Fixed by adding a `convertToJpeg` step in `fetch-thumbnail.js` for the direct-URL fallthrough path (non-YouTube, non-SDI URLs): images are now resized to a 1920px maximum dimension and saved as JPEG at 90% quality, keeping all board images well under the 2MB limit. `post-social-content.js` was updated to pass `imageBuffer` from the thumbnail fetch through to `postToMicroblog` so both fixes work together. Added `sharp` as a dependency for the JPEG conversion.
- (2026-06-16) Corrected the e2e dispatch pipeline test to accept the social guard firing as a valid outcome. The guard detects that social content was already posted today and skips dispatch — this is intended behavior, not a failure. Removed the masking mechanism (which temporarily rewrote spreadsheet dates to force a dispatch) — it was brittle, mutated real data, and still failed in production on 2026-06-15. Updated the two affected assertions to pass when either a dispatch occurs or the guard correctly skips.
- (2026-06-15) Dispatcher now skips platforms that already have a post URL when retrying a partially-failed row. Previously, resetting a row to pending after partial success (e.g., LinkedIn failed, Bluesky/Mastodon succeeded) would re-post to all platforms in the `platforms` column — including ones that had already posted successfully — creating duplicate posts. Each platform gate in `dispatchPost` now checks the corresponding URL column (Bluesky: `bskyPostUrl`, Mastodon: `mastodonPostUrl`, LinkedIn: `linkedinPostUrl`, micro.blog: `microblogPostUrl`) and skips the platform if a URL is already present. Rows where all platforms are already posted are correctly marked `posted` rather than `failed`.
